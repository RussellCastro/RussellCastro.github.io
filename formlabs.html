Problem

Each printer product had its own firmware implementation. When the product line grew from 2 to 5+ printers, features were being duplicated across codebases. A motor move task written for Printer A would be copy-pasted and adapted for Printer B, leading to divergent behavior, duplicated bugs, and high maintenance cost.

Before: Duplicated per product

// Printer A firmware class PrinterA_MotorMove : public Task { TaskResult Run(PrinterA_Context& ctx) { // 200 lines of motor control logic // mixed with hardware-specific SPI calls } }; // Printer B firmware (copy-pasted, diverged) class PrinterB_MotorMove : public Task { TaskResult Run(PrinterB_Context& ctx) { // Same logic, different hardware calls // Bug fixes applied here but not in A } };

After: Shared task, product-specific hardware

// Shared firmware (write once) class MotorMove : public SharedTask { TaskResult Run(Context& ctx) { auto motor = ctx.Get<I_MotorSubsystem>(); return motor->MoveRelative(axis, distance); } }; // Each product just implements the interface class PrinterA_Motor : public I_MotorHardware { /*...*/ }; class PrinterB_Motor : public I_MotorHardware { /*...*/ };

Architecture

Three-layer abstraction: Tasks define behavior, Subsystems manage state and coordination, and Hardware interfaces abstract the physical layer. Each product implements only the hardware layer, inheriting all shared tasks and subsystem logic.

Abstraction Layers

Shared Tasks Portable operations (e.g., RelativeMove, ReadSensor, NFC_Authenticate)
▼ calls
Shared Subsystems State management, motion planning, PID control, coordination logic
▼ calls
Abstract Hardware Interfaces Contract: I_MotorHardware, I_SensorHardware, I_NFCHardware, ...
▼ implemented by
Product-Specific Hardware Drivers SPI/I2C/protobuf communication with actual MCU hardware
Tasks never access hardware directly. Subsystems hold state. Hardware classes are stateless drivers.

Integration Pattern

// Product startup sequence wires shared subsystems to product-specific hardware class MotionSubsystemInit : public Task { TaskResult Run(Context& ctx) { // 1. Alias product hardware as the shared interface type auto hw = ctx.AliasResource<ProductHardware, I_MotorHardware>(); // 2. Create shared subsystem with hardware interface auto subsystem = std::make_unique<MotionSubsystem>(hw); // 3. Register in context -- all shared tasks can now use it ctx.StoreResource<I_MotionSubsystem>(std::move(subsystem)); } }; // After init, shared tasks like RelativeMove, HomeAxis, SyncMotors // work on ANY product that registered a MotionSubsystem.

Design Principles

Tasks should not access hardware

  • Tasks call subsystem methods
  • Keeps tasks portable across products
  • Enables unit testing without hardware

Hardware stores zero state

  • Hardware classes are stateless drivers
  • Translate commands, relay responses
  • Interchangeable per product

Subsystems own the logic

  • State machines, PID loops, coordination
  • Multi-hardware orchestration
  • Shared across all products

Init tasks are the glue

  • Each product wires its hardware to shared interfaces
  • One-time setup at startup
  • Only product-specific code needed

Impact

Before shared tasks

Adding a feature to N products required N implementations. A bug fix in one product might not propagate to others. New products required months of firmware bring-up copying and adapting existing code.

After shared tasks

  • New product bring-up reduced to implementing hardware interfaces (days instead of months for supported subsystems)
  • Bug fixes propagate to all products automatically
  • 12 subsystems shared across 4 product lines
  • Established conventions (namespace layout, error subsystem imports, CMake patterns) adopted team-wide
  • Wrote internal documentation used for onboarding new embedded engineers
