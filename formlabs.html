Problem

To bring up new printer products faster, the firmware platform needed a shared abstraction layer instead of per-product implementations. A motor move or NFC read written for Printer A would be copy-pasted and adapted for Printer B, leading to divergent behavior, duplicated bugs, and high maintenance cost.

Architecture

Actions define behavior, logic layers manage state and coordination, and Hardware drivers abstract the physical layer. Each product implements only the hardware driver, inheriting all shared actions and logic.

Integration Pattern

At product startup each printer wires its hardware adapters into the shared logic layer. After that, every shared operation works on the product without further changes.

  1. Product hardware is exposed as the shared driver type expected by the logic layer.
  2. The shared logic is constructed against that driver, picking up all of its behavior.
  3. The shared logic is registered against the running printer so shared operations can resolve it.
  4. Any operation in the shared layer now works on this product with no further changes.

Impact

Before shared Actions

Adding a feature to N products required N implementations. A bug fix in one product might not propagate to others. New products required months of firmware bring-up copying and adapting existing code.

After shared Actions

  • Potential future product bring-up reduced to implementing hardware interfaces (days instead of months for supported domains)
  • Bug fixes propagate to all products automatically
  • Major logic modules shared across flagship product lines
  • Established conventions (namespace layout, error-handling imports, CMake patterns) adopted team-wide
  • Wrote internal documentation used for onboarding new embedded engineers

Problem

Callback Spaghetti

Request handlers connected to response slots via signals, with manual correlation of which response belonged to which request.

Dead Code Paths

Unused callback hooks accumulated as the code evolved. No compiler warnings because slots are connected at runtime.

Error Swallowing

Failure paths often silently dropped errors because the callback chain didn't propagate them consistently.

Solution: Future-Promise Chains

Replaced signal/slot callback patterns with future-promise chains. Each async operation returns a future that the caller can chain, inspect, or await.

Before: Signals and Slots

  1. Caller emits a signal carrying a manually-created identifier.
  2. Signal bus broadcasts to every connected slot in the system.
  3. Each slot must filter by identifier to find its own match.
  4. Matching slot fires another signal back toward the caller.
  5. Errors often slip through the gap and never reach the caller.

Implicit wiring. Manual correlation. No compile-time guarantees.

After: Future-Promise Chain

  1. Caller invokes the operation and receives a handle.
  2. Caller attaches a continuation to the handle.
  3. When the operation completes, the result flows through the continuation.
  4. Errors are part of the handle and propagate to the caller naturally.

Explicit chain. Errors are first-class. No manual identifier tracking.
